# 性暴力幸存者故事 Agent —— MVP 产品设计

## 1. 项目概述

### 1.1 目标

将 9 位遭遇性暴力的嘉宾的采访文稿，结构化为 Agent 的 prompt，由 Agent 扮演嘉宾角色；用户通过对话了解嘉宾的故事，达到「可被看见、可被理解」的传播与共情目标。

### 1.2 核心价值

- **幸存者**：故事被尊重地保留与呈现，不二次消费创伤。
- **公众**：通过安全、可控的对话了解幸存者经历，促进理解与反思。
- **产品**：用 Agent 承载结构化故事，在边界清晰的前提下支持「对话式了解」。

### 1.3 重要前提

- 内容涉及性暴力，需**创伤知情（trauma-informed）**设计。
- 嘉宾需**知情同意**：明确同意以 Agent 形式呈现自己的故事及使用范围。
- 产品定位为**故事了解与共情**，不替代专业心理支持或法律咨询。

---

## 2. MVP 范围

### 2.1 在 MVP 内（Must Have）

| 能力 | 说明 |
|------|------|
| 采访文稿结构化 | 将每份采访转为：背景设定、关键经历节点、态度与观点、可引用原话、边界（不回答/不演绎的内容） |
| 单嘉宾 Agent | 一个 Agent 对应一位嘉宾，prompt 由上述结构化内容生成，行为可预期、不即兴编造 |
| 对话式了解故事 | 用户与 Agent 对话，Agent 仅基于「已授权故事」回答，可引导至关键经历与感受 |
| 嘉宾列表与选择 | 首页/列表展示 9 位嘉宾（可匿名/化名+简短介绍），用户选择一位进入对话 |
| 触发警告与资源 | 进入前/会话中的触发警告（trigger warning），以及固定入口提供心理支持/求助资源（如热线、机构） |
| 基础伦理与安全 | 不生成未在文稿中的具体创伤细节；不扮演施害者；敏感问题有统一兜底回复（如建议寻求专业帮助） |

### 2.2 不在 MVP 内（Later）

- 多轮记忆与长期画像、个性化推荐。
- 用户生成内容（UGC）、评论、点赞。
- 多语言、无障碍增强、数据统计看板。
- 嘉宾或管理员后台编辑 prompt/上下架。

## 3. 信息架构（IA）

```
首页
├── 触发警告 + 同意进入
├── 支持资源入口（常驻）
└── 嘉宾列表（9 位）
    └── 嘉宾卡片 [化名/匿名, 一句话介绍, 可选头像/剪影]
        └── 进入对话
            ├── 再次触发警告（可选）
            └── 与「该嘉宾」Agent 对话
                └── 会话内支持资源入口
```

## 4. 核心数据与流程

### 4.1 从采访到 Agent Prompt 的流水线

```
采访文稿（原始）
    → 结构化抽取（人工/半自动）
        → 嘉宾档案：
            - 基础设定（称呼、可公开背景）
            - 经历节点（时间/主题，不写细节可写“某类经历”）
            - 态度与观点（对司法、对舆论、对康复等）
            - 可引用原话（逐条，标注情境）
            - 边界清单（禁止演绎、禁止具体化、不回答的问题类型）
    → Prompt 模板填充
        → 系统 prompt：角色设定 + 边界规则 + 求助资源话术
        → 可选 few-shot：标准问与标准答
    → 部署为「单嘉宾 Agent」
```

### 4.2 对话边界规则（必须落实）

- 仅使用「嘉宾档案」内信息回答；对档案外问题回答「这不在我分享的范围内」或引导回已分享内容。
- 禁止生成：具体性暴力细节、施害者视角、对未发生事件的虚构。
- 若用户表露自身创伤或危机：回复固定话术，引导至热线/专业资源，不进行咨询或诊断。

### 4.3 用户侧流程

1. 进入站点 → 阅读触发警告与支持资源 → 同意后进入。
2. 浏览嘉宾列表 → 选择一位 → 可选再确认触发警告。
3. 与 Agent 对话：提问、追问，Agent 以该嘉宾口吻在边界内回答。
4. 随时可点击「支持资源」离开或寻求帮助。

## 5. 伦理与安全

### 5.1 知情同意

- 每位嘉宾签署/确认：内容用途、以 Agent 形式呈现、可随时撤回或修改范围。
- 书面约定：不超出采访与约定范围生成内容。

### 5.2 内容与行为边界

- 所有回复可追溯至「结构化档案」或明确标注「未在分享范围内」。
- 定期人工抽检对话记录，发现越界即调整 prompt/规则。

### 5.3 用户安全

- 醒目、可重复触达的**心理支持/危机资源**（如 24 小时热线、本地服务机构）。
- 触发警告覆盖：性暴力、创伤、自杀自伤等；文案由具备相关经验的人审核。

### 5.4 隐私与数据

- 对话记录：明确告知是否被存储、用途（仅改进产品/安全审计）、保留期限。
- MVP 建议：可仅做会话内上下文，不持久化用户身份与完整历史，以降低风险。

## 6. 技术方案概要

### 6.1 角色划分

| 模块 | 职责 |
|------|------|
| 内容生产 | 采访稿 → 结构化嘉宾档案（JSON/Markdown）→ 人工审核 |
| Prompt 工程 | 档案 → 系统 prompt + 边界规则 + 兜底话术 |
| Agent 运行时 | 调用 LLM，严格 system prompt，可选 RAG 检索「可引用原话」 |
| 前端 | 静态页 + 嘉宾列表 + 单页对话（选嘉宾 → 对话 UI + 支持资源入口） |
| 合规与安全 | 触发警告、同意记录、支持资源配置、日志与抽检 |

### 6.2 技术选型（建议）

- **Agent**：现有 LLM API + 强约束 system prompt；若本仓库为 echo-agents，可复用既有 agent 框架，将「嘉宾」建模为一种 agent 类型。
- **结构化存储**：每位嘉宾一个配置（档案 + prompt 模板），版本可追溯。
- **前端**：轻量 SPA 或静态站 + 对话组件即可；优先可访问性与加载速度。
- **部署**：静态前端 + 无状态 API（对话接口），便于审核与合规。

### 6.3 与 echo-agents 的衔接（若适用）

- 将「嘉宾 Agent」定义为一种 agent 类型，配置来自「嘉宾档案 + prompt 模板」。
- 对话 API 仅暴露「选择嘉宾 → 发送消息 → 流式/非流式回复」，不暴露通用 open-ended 能力。
- 日志与审计可复用现有 agent 调用日志，增加「嘉宾 ID、会话 ID、是否触发兜底」等字段。

## 7. MVP 里程碑（建议）

| 阶段 | 产出 | 验收标准 |
|------|------|----------|
| M0 合规与内容 | 知情同意流程、支持资源清单、触发警告文案 | 法务/伦理过目；至少 1 位嘉宾完整档案与审核通过 |
| M1 单嘉宾跑通 | 1 位嘉宾的结构化档案 + prompt + Agent 对话 | 边界内回答稳定；越界与危机话术正确触发 |
| M2 九嘉宾上线 | 9 位嘉宾全部上线、列表与选择、统一支持资源与警告 | 所有嘉宾可对话、无越界案例（抽检） |
| M3 发布与收尾 | 小范围发布、使用说明、反馈收集与一次 prompt 迭代 | 至少一轮真实用户反馈并完成一次改进 |

---

## 8. 成功指标（MVP）

- **安全**：零次「越界生成具体创伤细节/施害者视角」；危机话术 100% 按设计触发（抽检）。
- **体验**：用户能通过对话理解该嘉宾的故事主线与态度；完成率/停留时长可作为次要指标。
- **合规**：9 位嘉宾知情同意齐全；触发警告与支持资源可见、可点。

---

## 9. 文档与迭代

- 维护「嘉宾档案规范」与「Prompt 模板」文档，便于后续嘉宾扩展或协作。
- 每次新增/修改嘉宾或 prompt，需经过「边界清单」与「兜底话术」复核。
- 本 MVP 设计可根据首次用户反馈与伦理评审结果做小幅修订，再考虑迭代功能（如 M2 之后的多轮记忆、多语言等）。
